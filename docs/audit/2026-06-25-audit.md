# cmdit security audit ‚Äî 2026-06-25

> The v1.0 security-audit pass. cmdit parses **untrusted process input** (argv and,
> via `cmdit_env`, environment variables), in a language with **no bounds checking**
> (raw `load8`/`load64`/`store8`/`store64` over `alloc`'d memory). This audit asks
> one question: can attacker-controlled argv/env drive an out-of-bounds access, an
> overflow, or a NULL dereference?

**Result:** no attacker-reachable memory-safety defect. Two low-severity correctness
issues were found and **fixed in this cut**; the remaining items are info-level and
accepted with rationale below.

## Method

An adversarial multi-agent review (independent finders ‚Üí per-finding verification
that re-traced each claim and refuted false positives) over the entire library
surface (`src/cmdit.cyr`), with argv and env treated as fully attacker-controlled.
Every memory-safety claim on an attacker-reachable path was **refuted** under
verification (the buffers are bounds-checked before store, the comparisons are
length-bounded, the token scan is NUL-checked byte-by-byte). What survived is below.

## Threat model

- **Attacker-controlled:** the full argv vector (count, each token's bytes/length)
  and any environment variable named by `cmdit_env`.
- **Program-controlled (NOT an attacker primitive):** the flag/verb **index** passed
  to `cmdit_get_*` / accessors. Indices are the return value of `cmdit_bool`/`_int`/
  `_str`/`_enum`/`_verb` at registration time ‚Äî chosen by the program, never derived
  from argv/env. (The audit still hardened the accessors against a program bug ‚Äî an
  *ignored* `-1` registration return ‚Äî see SEC-2; that is robustness, not an attacker
  surface.)
- **Trust boundary:** registered flag names, choice sets, env-var names, and verb
  names are program-supplied constants (trusted). Argv tokens and env values are not.

## Findings

| ID | Severity | Title | Status |
|----|----------|-------|--------|
| SEC-1 | Low | `_cmdit_parse_int` silently wrapped on overflow (range bypass) | **Fixed** |
| SEC-2 | Low | Typed getters did not bounds-check `idx` | **Fixed** |
| SEC-3 | Info | Positional / repeat lists drop silently past their caps | Accepted |
| SEC-4 | Info | `--=` can match a flag registered with an empty long name | Accepted (misconfig) |
| SEC-5 | Info | Defensive: bounded comparisons & NUL-checked token scan are sound | Verified-good |

### SEC-1 ‚Äî int overflow silently wrapped (Low) ‚Äî FIXED

`_cmdit_parse_int` accumulated `n = n*10 + d` with no overflow check, so a long
all-digit argv/env token (e.g. `--width 18446744073709551617`, 2‚Å∂‚Å¥+1) was accepted
as "valid" and **silently wrapped modulo 2‚Å∂‚Å¥** ‚Äî wrapping to `1`, which then *passed*
a `cmdit_range(1, 4096)` bound. This violated the documented strict-parse contract
and handed consumers a wrapped value usable in a downstream `alloc`/count.

It is an i64 wrap, not a memory-safety bug (cmdit never indexes with the value), but
the range-bypass made it worth fixing. **Fix:** reject before the multiply when
`n > (INT64_MAX ‚àí d) / 10`, leaving `_cmdit_pint_ok = 0` ‚Üí `CMDIT_ERR_BAD_INT`. The
largest valid i64 (`9223372036854775807`) still parses exactly; `2‚Å∂‚Å¥+1` is now
rejected. Because the env fallback shares `_cmdit_parse_int`, an overflowing env
value is now non-fatally rejected too. Covered by the test group *"int overflow
rejected, not wrapped (no range bypass)"*.

### SEC-2 ‚Äî getters did not bounds-check idx (Low) ‚Äî FIXED

`cmdit_get_bool/int/str/enum` and `cmdit_seen` called `_cmdit_entry(h, idx)` with no
range check, while `cmdit_positional` / `cmdit_repeat_get` already guarded. A program
that **ignored a `-1` registration return** (the table-full sentinel) and then called
`cmdit_get_int(h, -1)` would read 112 bytes out of bounds. Not an attacker primitive
(idx is program-controlled), but an easy footgun to freeze into the API. **Fix:** a
shared `_cmdit_idx_ok(h, idx)` guard (`0 ‚â§ idx < count`) on all five accessors; a bad
idx returns the type's zero sentinel (`0`, or `-1` for `get_enum`) instead of reading
out of bounds. Covered by *"getters: negative / out-of-range idx ‚Üí safe sentinel"*.

### SEC-3 ‚Äî silent cap drops (Info) ‚Äî ACCEPTED

Positionals (`CMDIT_POS_MAX` = 128), the per-flag repeat list (`CMDIT_REPEAT_MAX` =
64), the flag table (`CMDIT_FLAGS_MAX` = 64) and the verb table (`CMDIT_VERB_MAX` =
128) all enforce their cap **before** the store (`if (count >= cap) return ‚Ä¶;`), so an
over-cap argv writes nothing out of bounds ‚Äî it is **memory-safe**. The behavior is
*lossy* (extra positionals/repeats are dropped; over-cap registrations return `-1`),
which is the intended, documented contract (same drop-past-cap rule as the stdlib
parser). Accepted as-is; the registration caps returning `-1` are now tested
(*"registering past CMDIT_FLAGS_MAX / CMDIT_VERB_MAX returns -1"*). A consumer needing
a hard error on overflow checks the `-1` / count itself.

### SEC-4 ‚Äî empty-long-name match via `--=` (Info) ‚Äî ACCEPTED

A token `--=value` computes `name_len = 0` and `_cmdit_find_long` would match a flag
**registered with an empty (`""`/0-length) long name**. This requires the *program*
to register a degenerate empty-named flag ‚Äî a misconfiguration cmdit does not produce
(`cmdit_new` and every demo pass real names or `0`, and a `0` long name is skipped by
the `ep_name != 0` guard). No attacker can induce a match against a normally-named
flag (the length-checked compare rejects prefixes ‚Äî see the test *"a prefix of a valid
choice is rejected"* and the long-name length check). Accepted as a
consumer-misconfiguration non-issue; documented so it is a known, bounded behavior.

### SEC-5 ‚Äî defensive properties (Info) ‚Äî VERIFIED-GOOD

The audit confirmed three load-bearing safety properties, recorded here so a future
change does not regress them:

- **Length-bounded comparisons.** `_cmdit_streq_bounded(a, b, len)` and
  `_cmdit_cstr_eq(a, b)` are always called with the **trusted, NUL-terminated**
  operand (`b` = a registered name) as the loop's bounding side, and the attacker
  slice as `a`. Neither walks past a NUL on the untrusted side. *Invariant: keep the
  trusted operand as the bound.*
- **NUL-checked token scan.** `cmdit_parse_argv` / `cmdit_dispatch_argv` /
  `_cmdit_disp_find` only ever read `tok+1`, `tok+2`, ‚Ä¶ **after** proving the prior
  byte is non-NUL (a `tok == 0` skip, then `load8(tok)`, then `load8(tok+1)` guarded
  by the previous being `45`/non-zero). A malformed short/empty token cannot over-read.
- **Env / NULL guards.** `_cmdit_apply_env` null-checks `getenv` before any use,
  filters set-but-empty for non-bool flags, and is non-fatal on a bad value;
  `cmdit_basename(0)` and the accessors handle the zero pointer.

## Conclusion

cmdit is memory-safe against untrusted argv and env on every reachable path. The two
low-severity correctness issues (SEC-1 overflow range-bypass, SEC-2 unguarded getter
idx) are fixed and tested in this cut; the info-level items are accepted with the
rationale above. This satisfies the v1.0 security-audit gate.
